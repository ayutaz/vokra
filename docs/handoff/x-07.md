## 1. crates.io (X-07-T14)

The publish set is the **15-crate topological closure** of `vokra-capi` +
`vokra-cli` (derived by `tools/release/crates_publish_order.py`; run it to see
the exact list + order). Owner steps, once:

1. **Reserve the 15 crate names** on crates.io (`vokra-core`, `vokra-cli`, the
   6 backend crates, etc. — `python3 tools/release/crates_publish_order.py`
   prints them). crates.io names are first-come.
2. **Add `secrets.CRATES_IO_TOKEN`** (repo secret). `crates-io-publish` is gated
   on it — absent = honest `::notice::` skip, so nothing publishes accidentally.
3. **Decide first-publish timing**: publish on an rc tag (CD auto), or keep
   dry-run only for now. The initial version is `0.1.0-alpha.0`; align it with
   the release tag semver before the first real publish (crates.io is immutable).

**Chicken-and-egg to expect** (recorded in the ADR): `cargo publish --dry-run`
for downstream crates cannot fully pass before the first real publish (it
resolves version deps against the live index). The dry-run job therefore gates
on `cargo package --list` for all 15 crates + a real dry-run of the leaf
(`vokra-core`); the downstream chain validates itself during the incremental
topological publish (`crates-io-publish`), which cargo sequences correctly.

**X-09b unblock**: crates.io publish is the precondition for docs.rs auto-doc
(docs.rs builds published crates → the API reference X-09b wants). `documentation`
is intentionally unset so crates.io auto-links each crate to `docs.rs/<crate>`.
Hand this to X-09b after the first publish.

## 2. Package-manager registration (X-07-T33)

CC generates the manifests via `python3 tools/release/gen_package_managers.py
--version <v> --out-dir <dir> [--assets-dir <release-assets>]` from templates in
`packaging/`. Owner registers each (accounts + PRs):

| manager | generated | owner action |
|---------|-----------|--------------|
| Homebrew | `packaging/homebrew/vokra.rb` (source build) | create a tap, add the formula |
| winget | `packaging/winget/*.yaml` | PR to `microsoft/winget-pkgs` |
| vcpkg | `packaging/vcpkg/ports/vokra/` (source) | PR to `microsoft/vcpkg` |
| Debian | `packaging/debian/` | sponsor-upload (mangle `-rc.N` → `~rc.N`) |

Homebrew + vcpkg are source-tarball based (checksums reachable from a tag).
winget + Debian reference the T32-gated desktop binaries — their checksum slots
render as explicit `*-PENDING-RELEASE` until §4 is enabled and re-rendered.

## 3. Release cadence (X-07-T23)

`.github/workflows/release-cadence.yml` (Mon 09:00 UTC + dispatch) measures days
since the last semver tag and reports **due** (>= 28 days) to the run's step
summary — it **never** tags (owner-only). When due, issue the `vX.Y.Z` tag to
fire `release.yml`. Before tagging, roll the changelog:
`python3 scripts/release/roll-changelog.py --version X.Y.Z` (renames
`[Unreleased]` → the release section; `release-notes` then extracts it).

The cadence judgment (schema `vokra.release-cadence.v1`, uploaded as the
`release-cadence-judgment` artifact) is the SoT **M5-12-T06 consumes** for GA
DoD item 3 (release-train stable operation): data source = `git-tags:refs/tags/v*`,
28-day **closed** boundary (`>=`), `cadence_established` = ≥2 releases with every
interval ≤28d. With 0 releases today it honestly reports "not established".
