# M4-15 handoff — CPU + Vulkan-only build target (owner gates / follow-ups)

M4-15 の CC 実装分（feature plumbing / `build-target-vulkan-only` CI job /
NOTICE variant 導出 / SPDX 2.3 SBOM / 非リンク scanner / 表記 gate）は land
済み。設計判断の全文は ADR M4-15（`docs/adr/M4-15-cpu-vulkan-build-target.md`、
gitignore local）。以下は **owner 側 gate と follow-up** の記録。

## (a) T10 — SBOM reproducible-build verify（owner 専任）

同一入力（同一 commit + 同一 toolchain + 同一 feature 選択）から SBOM が
byte-identical に再生成されることを owner 環境で確認する。

```sh
python3 scripts/sbom/generate_spdx.py \
  --package vokra-capi --no-default-features --features vulkan \
  --doc-name vokra-capi-cpu-vulkan-only --output /tmp/sbom-a.json
python3 scripts/sbom/generate_spdx.py \
  --package vokra-capi --no-default-features --features vulkan \
  --doc-name vokra-capi-cpu-vulkan-only --output /tmp/sbom-b.json
cmp /tmp/sbom-a.json /tmp/sbom-b.json && echo bit-identical
```

- 決定性の設計: `created` は `SOURCE_DATE_EPOCH`（未設定時 HEAD commit
  timestamp）、`documentNamespace` は git SHA 由来（UUID 不使用）、
  sorted keys / sorted packages、ローカルパス非混入。wall clock は使わない。
- CI の `sbom self-test`（`scripts/sbom/test-generate-spdx.sh` b4/b5）が
  同一 runner 内 2 回生成の byte-identical を常時検証済み。T10 は **owner
  のリリース環境**（別マシン / 別 checkout）での再現確認が本質。
- 差分が出た場合は原因（環境依存フィールド）を記録して CC に fixup 依頼
  （fabricated pass 禁止 = 一致の根拠 or 差分の記録を残す）。
- 任意: 外部 SPDX validator での追検証。

## (b) M4-13 owner gate — 本 build target の「動かない SKU 禁止」判定

本 WP は build 面（build + clippy + 非デバイス test + SBOM + NOTICE +
scanner）を先行 land した。**Vulkan で実加速すること**の証明は M4-13 の
exit gate であり、以下が揃って初めて成立する:

1. precompiled `.spv` の commit（M4-13-T16、owner。本 handoff 時点で
   `crates/vokra-backend-vulkan/kernels/precompiled/` は README のみ）
2. `gpu-vulkan-parity.yml`（lavapipe）green + Android 実機 soak

それまで本 target の Vulkan 指定は明示エラー（`UnsupportedOp` /
`BackendUnavailable`、FR-EX-08 = silent CPU fallback 無し）で、e2e は CPU
経路で成立する。

## (c) `build-target-vulkan-only` job の required-check 昇格

advisory job として land 済み。`main` の branch protection required check
への昇格は、M4-13 Vulkan parity が安定 green になった後の owner 判断
（`gpu-vulkan-parity.yml` の「数週連続 green 後に promote」方針と平仄）。

## (d) release.yml 統合 follow-up — SBOM のリリース同梱

NFR-MT-03 + NFR-MT-08 + X-07 より、リリース artifact に SBOM を同梱する。
本 WP は CI 側で artifact（cdylib + LICENSE + NOTICE variant + SPDX JSON、
名称 `vokra-cpu-vulkan-only-linux`）の生成 + upload まで実装済み。
release.yml への添付（`godot-package-release` job と同様の CI artifact
reassemble）は M4 リリース train 統合タイミングの follow-up。

## (e) 表記の分離（2026-07-14 hard gate (f)）と SKU 検出 API

- 本 build target に関する文書 / release notes / artifact 名は中立表記
  「CPU + Vulkan-only build target」で統一する。市場ポジショニング表現は
  M5-08/M5-11 残置（禁止語リストと機械 gate =
  `scripts/compliance/check-no-market-claims.sh`、CI 常設。M5 側で解禁する
  際は gate の走査対象を意識的に再設定する）。
- SKU 検出の実行時 C API は追加しない = M5-13 C ABI 凍結の対象外、将来
  必要なら v1.1 additive（`docs/handoff/m4-12.md` §(e) 整合）。ビルド種別は
  ビルド時 metadata（SBOM / NOTICE variant / artifact 名）で識別する。

## (f) ADR の保全（gitignore local）

ADR M4-15 は worktree 内 `docs/adr/M4-15-cpu-vulkan-build-target.md` に
作成済み（worktree sandbox が main checkout への直接 Write を禁止するため
— M4-14 ADR と同じ事情）。**worktree cleanup 前に main checkout の
`docs/adr/` へコピーして保全すること。**
