# M4-02 handoff — Unity WebGL delivery requirements + owner items (M4-11 input)

WP: M4-02 (Unity WebGL target for `com.vokra.unity`). CC-side automation is
complete; this file fixes (a) the web delivery requirements the demo needs,
(b) the exact owner follow-ups, so the M4-11 owner task (real-browser test +
CDN selection) can start from this document alone.

Design SoT: ADR M4-02 (`docs/adr/M4-02-unity-webgl-link.md`, gitignored
local doc — copy travels with the merge notes). Empirical basis: the
staticlib + `vokra_session_create_from_bytes` chain PASSES the full Silero
VAD run under node with BOTH Unity-matching Emscriptens (3.1.8 = Unity
2022.3, 3.1.38 = Unity 6); the file-path loader fails loudly there
(rust-std stat ABI skew) — hence the bytes-based model path.

## 1. What ships in the package

- `Plugins/WebGL/libvokra.a` — wasm32-unknown-emscripten staticlib,
  `-C panic=abort`, CPU-only (`--no-default-features --features cpu`),
  simd128 OFF (works regardless of Unity's "WebAssembly SIMD" setting).
  Assembled by CD (`ci.yml` `unity-package` / `release.yml`
  `unity-package-release`); never committed.
- C# surface: `VokraSession.CreateFromBytes` +
  `VokraAndroidAssets.ReadBytesAsync` (model path),
  `EnsureLocalCopy` throws `NotSupportedException` on WebGL (deadlock trap
  fixed), `EnsureLocalCopyAsync` = managed-IO materialisation only.
- Demo: `Samples~/VadAsrTts` auto-runs on WebGL and prints
  `[vokra-demo] WEBGL PASS` / `[vokra-demo] WEBGL FAIL` to the browser
  console (the machine-checkable smoke contract).

## 2. Web delivery requirements (serve the Unity WebGL build)

- **MIME**: `.wasm` → `application/wasm` (required for streaming compile);
  `.js` → `text/javascript`; Unity `.data` → `application/octet-stream`.
- **Compression**: match the server `Content-Encoding` to the Unity build's
  compression setting (Brotli/gzip), or enable Unity's "Decompression
  Fallback" when the server cannot set the headers. The nightly smoke serves
  uncompressed builds (Python stdlib server) — CDN compression is an
  M4-11-side choice.
- **COOP/COEP**: NOT required for the current baseline — threads are OFF
  (single-threaded WASM; the production C ABI path spawns no threads —
  machine-gated by `scripts/check-capi-thread-free.sh`) and the package `.a`
  uses no SharedArrayBuffer. The nightly smoke deliberately serves WITHOUT
  those headers, so a green run doubles as the empirical confirmation once
  UNITY_LICENSE exists. They become REQUIRED only when the WASM Threads
  opt-in lands (M4-01歩調の将来 opt-in; ADR M4-01 §3 の Worker+SAB 構成).
- **Model GGUFs**: NOT bundled (NFR-DS-04). The demo fetches
  `StreamingAssets/models/*.gguf` over HTTP at runtime
  (`fetch-demo-models.sh` stages them; Silero VAD = 2.1 MB committed
  fixture, Whisper base / piper voice = tens of MB). Serve with long-lived
  cache headers + content hashing on a real CDN; first load always pays the
  fetch. IndexedDB caching of fetched models is a follow-up (not
  implemented — the demo re-fetches per load).
- **Memory**: model bytes are copied once into the wasm heap
  (`CreateFromBytes`). Whisper base-scale models on WebGL need the Unity
  WebGL memory settings sized accordingly — real numbers are an owner
  measurement item (fabricating them here is prohibited).

## 3. CI structure (what runs when)

- Per-PR (`ci.yml` `unity-package`): staticlib build (rustc only, no emsdk),
  GPU-symbol audit, thread-free gate, `.meta`/package lints, tarball
  includes `Plugins/WebGL/libvokra.a` (asserted via `tar -tzf`).
- Nightly (`nightly-webgl.yml`):
  - `wasm-harness` leg — **runs without UNITY_LICENSE**: pinned-emsdk link
    of `tests/capi/smoke_vad_bytes.c` + node execution of the real VAD chain.
  - `unity-webgl-build` + browser smoke leg — license-gated; skipped with a
    `::warning::` until the secret exists (no fabricated pass). Chrome
    `--headless=new --enable-logging=stderr` captures the console; the job
    requires the PASS marker and fails on the FAIL marker.
- Release (`release.yml` `unity-package-release`): WebGL built in-job
  (fail-loud), 6-platform presence assert before packing.

## 4. Owner follow-ups (in order)

1. **UNITY_LICENSE provisioning** (M2-11 deferred, unchanged) → first
   `workflow_dispatch` of `nightly-webgl.yml`; expect: wasm-harness leg
   green immediately, Unity leg green after license.
2. **Real-browser check + CDN selection (M4-11)**: build the sample for
   WebGL from a WebGL-module-equipped Editor (local editors 6000.0.35f1 /
   6000.0.43f1 / 6000.1.1f1 lack the module today), serve per §2, confirm
   the PASS marker + audible TTS, record load times / memory. §2 is the
   requirements input for the CDN choice.
3. **Unity 2022.3 line check (optional)**: the node harness covers emcc
   3.1.8, but the nightly pins one Unity version; a 2022.3 Editor build is
   an owner spot-check if 2022.3 support must stay claimed.
4. **WebGPU-in-Unity-WebGL follow-up** (FR-BE-05 differentiation): blocked
   by architecture today — M4-01's glue needs `Atomics.wait` (worker) while
   Unity WebGL runs Vokra on the main thread. Re-check triggers: JSPI
   shipping in Safari/Firefox, or Unity WASM Threads adoption
   (ADR M4-02 §6). The WP completion criteria do not require WebGPU.

## 5. Known limitations (documented, not bugs)

- `vokra_asr_transcribe` / `vokra_tts_synthesize` block the tab for their
  duration (demo logs a notice; streaming poll is the primary path).
- A Rust panic is a wasm trap (`panic=abort`); `VOKRA_ERROR_PANIC` never
  fires on WebGL.
- `vokra_session_create_from_file` on WebGL fails loudly under Unity-era
  Emscripten (stat ABI skew) — by design the bytes API is the model path;
  the symbol is kept for future Emscripten/rustc convergence.
