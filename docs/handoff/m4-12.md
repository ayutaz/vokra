## (c) Post-v1.0 GA breaking-change rule

- Post-v1.0 GA, **breaking changes to `include/vokra.h` or the `vokra.*` GGUF schema require a major version bump (v2.0.0)** вЂ” no exceptions during minor/patch releases. ## (e) Semver anchor expansion вЂ” advisory documentation to land before M4-12 freeze

Sourced from the 2026-07-13 M4 scope-expansion mid-review (`docs/m4-scope-expansion-2026-07-13.md` В§жЋЁеҐЁг‚ўг‚Їг‚·гѓ§гѓі 5). These notes are advisory today and become authoritative at the v1.0 GA tag; landing them now buys owner headroom to add BIG-1гЂњBIG-10 candidate scope without triggering ABI regret at freeze time. All four items are documentation-only; no code changes are required at M3-16, only that M4-12 honor them when writing the freeze release notes / STABILITY block.

1. **HTTP/gRPC/WebSocket API is NOT part of the C ABI freeze.** `vokra-server` (M2-09) + a future Wyoming Protocol Server (BIG-9 candidate, per `docs/m4-scope-expansion-2026-07-13.md`) live at a separate governance tier вЂ” "**protocol-tracking / experimental**". Add a `## Non-C-ABI surface areas` section to `include/vokra.h` STABILITY block at M4-12 that names these and states they follow their upstream protocol's semver, not Vokra's. This prevents downstream users from treating a Wyoming Protocol breaking change (HA-community-driven, no back-compat guarantee) as a Vokra semver violation.

2. **Enums that may grow at v1.x need `#[non_exhaustive]` + a documented reservation range before freeze.** Two specific pre-declarations to write into the `## Reserved additions` section of `docs/abi-changelog.md`:
   - `IsaPath` enum (BIG-6 CPU ISA server tier expansion вЂ” AVX-512/VNNI/BF16/AVX-VNNI 256-bit/ARM64 dotprod/i8mm/bf16). Reserve variant slots for post-v1.0 additions of AMX (`Amx*`), SME (`Sme*`), and RVV Zvfh (`RvvZvfh*`) so those show up as variant additions (backward-compat OK) instead of enum-shape breaking changes.
   - FSQ codec API surface (BIG-5, `wavtokenizer_vq` + `xcodec2_fsq`). The `vokra.wavtokenizer.*` and `vokra.xcodec2.*` GGUF metadata schemas should be declared `EXPERIMENTAL` at freeze (like `M3-03 paged KV cache` was) so schema evolution is legal at minor bumps until the codec API stabilizes.
   - Matcha metadata (`vokra.matcha.*`, BIG-4) вЂ” same `EXPERIMENTAL` marker; landing Matcha-TTS at v1.0 doesn't require freezing the schema at the same tag.

3. **Delegate-selection API for NPUs (CoreML / QNN) is intentionally excluded from M4-12 semver anchor.** Per `docs/m4-scope-expansion-2026-07-13.md` В§е‰ЌеЂ’гЃ—г‚’жЋЁеҐЁгЃ—гЃЄгЃ„й …з›® the NPU delegates are M5-01/M5-02 hard-locked at M5, and the "delegate selection" API surface (whatever it ends up being вЂ” likely a `vokra_backend_kind_e` enum extension + a `vokra_session_bind_delegate` fn) is a **post-v1.0 additive** for M5 landing. Do NOT reserve variant slots or fn slots for delegate selection at M4-12; leave the surface entirely open so M5 implementation can iterate freely without pre-committing to an API shape before real-hardware bakeoff on ANE + Hexagon.

4. **`WatermarkConfig::backend_status` (Deferred / Active) transition is patch-release-allowed.** Currently the FR-CP-01 / FR-CP-02 config surface (M2-13) exposes a `WatermarkConfig` with a `backend_status` field that returns `Deferred` (per 2026-07-04 owner drop). BIG-8 in the scope-expansion doc proposes reviving runtime AudioSeal + C2PA embedding вЂ” which would flip `backend_status` from `Deferred` to `Active`. **At M4-12 freeze, add a clause to the STABILITY block that this specific transition is a patch-release (v1.x.y) allowed change**, not a minor bump requirement, so BIG-8 can land as a v1.0.x patch if the owner reversal happens post-freeze. Without this clause, an owner reversal after M4-12 would either force a v2.0 major bump (disproportionate for a documented-config transition) or a documentation regret at v1.0 GA.
