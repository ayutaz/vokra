## (c) Post-v1.0 GA breaking-change rule

- Post-v1.0 GA, **breaking changes to `include/vokra.h` or the `vokra.*` GGUF schema require a major version bump (v2.0.0)** — no exceptions during minor/patch releases. Prerelease permissiveness (v0.9.x free rename/remove) ends at the v1.0 tag; `docs/abi-changelog.md` §`Pre-1.0 policy (prerelease semver)` already states this policy.
- Additions that are backward-compatible in a C-linkage sense (new function, new `#[non_exhaustive]` enum variant, new opaque handle type) remain permitted at minor bumps.

## (e) Semver anchor expansion — advisory documentation to land before M4-12 freeze

Sourced from the 2026-07-13 M4 scope-expansion mid-review (`docs/m4-scope-expansion-2026-07-13.md` §推奨アクション 5). These notes are advisory today and become authoritative at the v1.0 GA tag; landing them now buys owner headroom to add BIG-1〜BIG-10 candidate scope without triggering ABI regret at freeze time. All four items are documentation-only; no code changes are required at M3-16, only that M4-12 honor them when writing the freeze release notes / STABILITY block.

1. **HTTP/gRPC/WebSocket API is NOT part of the C ABI freeze.** `vokra-server` (M2-09) + a future Wyoming Protocol Server (BIG-9 candidate, per `docs/m4-scope-expansion-2026-07-13.md`) live at a separate governance tier — "**protocol-tracking / experimental**". Add a `## Non-C-ABI surface areas` section to `include/vokra.h` STABILITY block at M4-12 that names these and states they follow their upstream protocol's semver, not Vokra's. This prevents downstream users from treating a Wyoming Protocol breaking change (HA-community-driven, no back-compat guarantee) as a Vokra semver violation.

2. **Enums that may grow at v1.x need `#[non_exhaustive]` + a documented reservation range before freeze.** Two specific pre-declarations to write into the `## Reserved additions` section of `docs/abi-changelog.md`:
   - `IsaPath` enum (BIG-6 CPU ISA server tier expansion — AVX-512/VNNI/BF16/AVX-VNNI 256-bit/ARM64 dotprod/i8mm/bf16). Reserve variant slots for post-v1.0 additions of AMX (`Amx*`), SME (`Sme*`), and RVV Zvfh (`RvvZvfh*`) so those show up as variant additions (backward-compat OK) instead of enum-shape breaking changes.
   - FSQ codec API surface (BIG-5, `wavtokenizer_vq` + `xcodec2_fsq`). The `vokra.wavtokenizer.*` and `vokra.xcodec2.*` GGUF metadata schemas should be declared `EXPERIMENTAL` at freeze (like `M3-03 paged KV cache` was) so schema evolution is legal at minor bumps until the codec API stabilizes.
   - Matcha metadata (`vokra.matcha.*`, BIG-4) — same `EXPERIMENTAL` marker; landing Matcha-TTS at v1.0 doesn't require freezing the schema at the same tag.

3. **Delegate-selection API for NPUs (CoreML / QNN) is intentionally excluded from M4-12 semver anchor.** Per `docs/m4-scope-expansion-2026-07-13.md` §前倒しを推奨しない項目 the NPU delegates are M5-01/M5-02 hard-locked at M5, and the "delegate selection" API surface (whatever it ends up being — likely a `vokra_backend_kind_e` enum extension + a `vokra_session_bind_delegate` fn) is a **post-v1.0 additive** for M5 landing. Do NOT reserve variant slots or fn slots for delegate selection at M4-12; leave the surface entirely open so M5 implementation can iterate freely without pre-committing to an API shape before real-hardware bakeoff on ANE + Hexagon.

4. **`WatermarkConfig::backend_status` (Deferred / Active) transition is patch-release-allowed.** Currently the FR-CP-01 / FR-CP-02 config surface (M2-13) exposes a `WatermarkConfig` with a `backend_status` field that returns `Deferred` (per 2026-07-04 owner drop). BIG-8 in the scope-expansion doc proposes reviving runtime AudioSeal + C2PA embedding — which would flip `backend_status` from `Deferred` to `Active`. **At M4-12 freeze, add a clause to the STABILITY block that this specific transition is a patch-release (v1.x.y) allowed change**, not a minor bump requirement, so BIG-8 can land as a v1.0.x patch if the owner reversal happens post-freeze. Without this clause, an owner reversal after M4-12 would either force a v2.0 major bump (disproportionate for a documented-config transition) or a documentation regret at v1.0 GA.

**Boundary reminder**: none of items (1)-(4) require code changes at M3-16. Item (2) is the only one requiring a text edit outside this handoff — the `## Reserved additions` section in `docs/abi-changelog.md`, which M3-16 already reserves as its edit surface. The freeze-executing WP (M5-13 after the 2026-07-14 reassignment, see (f)) owns the actual `include/vokra.h` STABILITY block rewrite and the CHANGELOG.md `[1.0.0]` release-note wording, and both must honor items (1), (3), (4) verbatim.

## (f) 2026-07-14 v-label reassignment #2 — freeze moves to M5-13, M4 becomes v1.0-rc

Owner decision 2026-07-14 (same day as, but after, the M4 scope-expansion ratification): **the entire scope through the former v2.0 (= M5) ships as v1.0**. Consequences for this handoff:

1. **New version ladder**: M3 = v0.9 (unchanged) / M4 = **v1.0-rc** (was v1.0 GA) / M5 = **v1.0 GA** (was v2.0 GA). v2.0 is pushed out of the current plan as a future major. Milestone IDs and WP scopes are unchanged.
2. **The IF-01 freeze fires in M5-13 (new WP), not M4-12.** The v1.0 GA tag — the freeze point named throughout sections (b)–(e) — is now the M5 close tag. All action items in (b) and the CI promotion in (d) transfer to M5-13 verbatim (with "v1.0 baseline" meaning the M5-close header). M4-12 is re-scoped to: snapshot `docs/abi/vokra.h.v1.0-rc-baseline.symbols` at the v1.0-rc tag, keep `scripts/abi-diff.sh` / `scripts/check-abi-changelog.sh` advisory, and keep the STABILITY block in its "NOT frozen (freeze at v1.0 GA)" form.
3. **The prerelease ABI policy window extends through the rc series.** v1.0-rc is a semver prerelease (`1.0.0-rc.N`); the `docs/abi-changelog.md` "Pre-1.0 policy (prerelease semver)" clauses (free rename/remove + mandatory changelog entry) stay in force until the v1.0 GA tag, exactly as written — the tag simply moved.
4. **Section (e)-3 is superseded in its motivation, kept in its instruction.** The NPU delegate API (M5-01/02) now lands *before* the freeze, so it can be **included in the frozen surface after the real-hardware bakeoff** — the original regret scenario (freezing before bakeoff) is resolved by the move rather than by exclusion. The instruction "do NOT reserve variant slots or fn slots for delegate selection at the rc tag" still holds for M4-12's baseline snapshot.
5. **Section (e)-2 EXPERIMENTAL markers** are applied at the (moved) freeze = M5-13 / v1.0 GA tag. Section (e)-4's patch-release clause now applies to post-GA patches (v1.0.x after M5 close); landing Watermark revival earlier during the rc window remains legal as a pre-freeze additive.
6. **Trade-off accepted by the owner**: the stable-ABI commitment to Unity/Godot integrators arrives at v1.0 GA (2027-07〜2028-01 estimate) instead of 2026-12〜2027-03. The rc tag still gives integrators a recorded, diffable baseline (advisory gate), just not a frozen one.
