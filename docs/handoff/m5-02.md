# M5-02 handoff — QNN backend (Qualcomm Hexagon NPU delegate)

**WP**: M5-02 (docs/milestones.md §9). **State**: CC scaffold slice landed
(SDK-free range). **Owner** tasks and the CC re-issue gate are below.

## What landed (CC, SDK-free — this branch)

- New crate `crates/vokra-backend-qnn` (raw dlopen loader + probe + `Backend`
  trait handle), zero external deps (NFR-DS-02). 4 modules (sys / probe /
  backend / lib), mirroring the CoreML sister delegate scaffold — **not** the
  metal 6-module layout: `context.rs` / `eval.rs` (QNN graph construction) are
  the re-issue wave's, not this slice's.
- `BackendKind::Qnn` variant (`#[non_exhaustive]`, additive) + rustdoc; the
  `Compute::for_backend` Qnn arm (probes first, then coverage → explicit
  `UnsupportedOp` / `BackendUnavailable`, never a silent CPU fall back);
  `covered_by_qnn` (all `false` in the scaffold) + a lock-step coverage test.
- `qnn` feature (default OFF) in `vokra-models` (forwards
  `vokra-backend-qnn/qnn`, the vulkan pattern); `--backend qnn` accepted by the
  CLI (`bench`/`run`); `deny.toml` bans QNN/Hexagon binding crates (distinct
  from the NNAPI ban — see below); build-only CI arm (`gpu-backends`, Linux).
- `third_party/QUALCOMM-QNN-NOTES.md` (5-point install-model record) +
  `docs/adr/M5-02-qnn-delegate.md` (**Proposed**, gitignored-local) + this file.
- **Rust surface only** — no new C ABI symbol (`include/vokra.h` unchanged; the
  abi-changelog records the Rust variant for the M5-13 freeze decision).

## The one honest constraint

No Qualcomm AI Engine Direct SDK is on the authoring host (Apple M1). Every QNN
**library name** (`libQnnHtp.so` / `QnnHtp.dll`), **symbol name**
(`QnnInterface_getProviders`), and **struct layout** below is from the WP
instruction, **not** first-hand verified. The raw FFI (`sys.rs`) + its
compile-time layout assert compile only on the Android/Linux/Windows CI arm, not
on macOS; the layout assert is a **self-consistency guard**, not a real-header
check. Everything else in the scaffold is verified on the authoring host (the
stub path) and on the Linux CI arm.

## NOT NNAPI (FR-BE-07) — do not conflate

QNN targets the Qualcomm **Hexagon NPU** through Qualcomm's own SDK. **NNAPI** —
the Android abstraction Google deprecated in Android 15 — is a different thing
and is **permanently unsupported**. Android's general GPU path is Vulkan
(M3-02); QNN is the Hexagon-NPU delegate. `deny.toml` bans both families, with a
comment marking the distinction.

## Owner tasks (T11, T12) — cannot be done by CC

1. **T11 — SDK acquisition + EULA + real-header layout verification.** Download
   the Qualcomm AI Engine Direct SDK, accept the Qualcomm license, and verify
   the compile-time layout asserts in `crates/vokra-backend-qnn/src/sys.rs`
   against the real SDK headers (size / align / field offsets — the same
   `clang`-probe verification M3-11's GDExtension did, except there is no header
   to probe until you install the SDK). Confirm the exact library SONAME / DLL
   name and the interface entry symbol spelling. **This is the gate** for the CC
   re-issue wave (below). Point the probe at your SDK's library during testing
   with `VOKRA_QNN_LIB=/path/to/libQnnHtp.so`.
2. **T12 — Snapdragon real-device bakeoff + 2× sign-off.** Once the
   re-issue wave lands, run the QNN backend e2e on a Snapdragon device, prove
   execution ran on the Hexagon NPU (not a CPU/GPU backend fallback), and judge
   NFR-PF-12 "2× speed-up" against the **M5-14-post CPU baseline**
   (`docs/bench-baselines/m5-14-final-2026-07-18/`). A miss is an honest
   negative (record it; do not loosen the target — the M3 "FA v2 SLOWER" record
   is the precedent). Decide whether to promote the delegate selector to the C
   ABI at M5-13.

## The CC re-issue wave (gated on owner T11 — NOT owner implementation work)

The WP completion condition ("QNN delegate, 2× faster, verified by bench")
requires a **functional** QNN backend, which this scaffold is not (every op is
`UnsupportedOp`). The implementation phase between this scaffold and T12 is a
**CC re-issue wave** whose acceptance criteria cannot be written until the SDK
headers are on disk (struct layout / symbol spelling / API-version negotiation
are UNKNOWN here). Once T11 is done, re-issue a CC wave to implement:

- `context.rs` / `eval.rs`: the `QnnGraph_create` → `addNode` → `finalize` →
  `execute` call bodies (raw dlopen FFI — CC-implementable once headers exist,
  the M2-03 CUDA Driver API + NVRTC precedent).
- The op-mapping table + the model-supply scheme decided in the ADR (case A
  runtime graph construction / case B offline QNN context binary).
- The ADR case-D execution-device readout (HTP vs CPU/GPU backend) so T12's 2×
  bench cannot accidentally measure a CPU/DSP fallback.
- `covered_by_qnn` per-op flips (the Vulkan T14…T22 lock-step pattern), and —
  only when a real `QnnContext` is constructible — the `Be::Qnn` variant +
  `backend_name` arm (adding them in the scaffold would be dead code that breaks
  `clippy -D warnings`).
- The honest NPU-execution parity bound (the M5-01-T20 pattern; **do not loosen
  a bound to make CI green**).

## M5-13 (C ABI freeze) note

A **C-level** QNN delegate selector is intentionally not exported in the v1.0-rc
window (same posture as CoreML — `include/vokra.h`, `docs/handoff/m4-12.md`). So
during v1.0-rc a C consumer cannot select QNN; the Rust surface
(`with_backend(BackendKind::Qnn)` / `vokra-cli --backend qnn`) is the only path.
Whether to promote the selector to the C ABI is an M5-13 decision after the
real-hardware bakeoff — bundled with the CoreML selector decision so both
delegates freeze together.

## Delegate commonisation (M5-01 ↔ M5-02) note

Both delegates (CoreML, QNN) share the scaffold shape: probe-first
`for_backend`, empty `covered_by_*`, no `make_backend` arm (a delegate executes
whole submodels, not per-op graph nodes), Rust-surface-only selector. The
difference to litigate in the re-issue waves: CoreML hands device placement to
the Apple runtime, whereas QNN builds the graph explicitly via `QnnGraph_*`. The
M5-01-T02 delegate-boundary ADR is the shared source of truth; M5-02 inherits it
and does not re-argue it.
