## (2) T08 upstream parity re-measurement (the highest-risk change)

The Silero forward's `exp` (sigmoid), `tanh` (LSTM cell/output) and `sqrt` (pseudo-STFT magnitude) were switched from `std` `f32::{exp,tanh,sqrt}` (libm) to the self-contained `vokra_vad_micro::scalar::{exp,tanh,sqrt}` (so the std and no_std builds are bit-identical). This moves the **std** build's VAD numbers off libm, so the existing upstream/ORT parity was re-measured on this machine. `max_abs_err` vs the ORT reference, `atol = 0.01` (NFR-QL-01), all legs:

| leg | BEFORE (std libm) | AFTER (shared scalar) | atol | margin |
|-----|-------------------|-----------------------|------|--------|
| 16k stage: stft_conv / magnitude / encoder | 0 / 0 / 4.88e-4 | 0 / 0 / 4.88e-4 | 0.01 | — |
| 8k stage: stft_conv / magnitude / encoder | 0 / 0 / 1.04e-5 | 0 / 0 / 1.04e-5 | 0.01 | — |
| 16k raw e2e | 7.893e-8 | 7.893e-8 | 0.01 | ~127000× |
| 16k ctx e2e | 2.056e-6 | 1.982e-6 | 0.01 | ~5000× |
| 8k raw e2e | 3.189e-6 | 3.234e-6 | 0.01 | ~3000× |
| 8k ctx e2e | 3.278e-7 | 5.513e-7 | 0.01 | ~18000× |
| jfk 16k ctx (real speech) | 6.139e-6 | 5.484e-6 | 0.01 | ~1800× |
| jfk 8k ctx (real speech) | 1.192e-6 | 1.431e-6 | 0.01 | ~7000× |

**Verdict: bound HELD; land.** The shift is a few-ULP transcendental change dominated by the accumulation-order difference vs ORT (some legs nudged up, some down). The `atol = 0.01` bound was **not** relaxed (red line #3) — it is unchanged from M0-05. The real-speech legs still detect exactly the 4 segments and peak `max_prob = 1.0000` at both rates.

**NFR-QL-05 confirmation (the `sqrt` route)**: `pseudo_stft.rs` is the self-declared NFR-QL-05 red line (`magnitude = sqrt(re²+im²)`, upstream-faithful). The **magnitude stage `max_abs_err` is `0.00e0` in BOTH before and after** — the Newton `sqrt` (a bounded few ULP off HW `vsqrt`) does not degrade the pseudo-STFT fidelity: the magnitude feeds a conv → ReLU → LSTM stack, none of which amplifies the deviation toward `atol`. So the Newton default (which keeps the crate `unsafe`-free, NFR-RL-07) is honest to land; an FP-armv8 `asm!("vsqrt.f32")` (IEEE-exact) remains an opt-in owner choice (§5, T18).

## (3) T12 memory budget (Cortex-M55 SRAM pass/fail)

**Static code** (`.text`, thumbv8m.main-none-eabihf, opt-level 2, debuginfo off, `llvm-size`; a **pre-link, no-dead-strip UPPER bound** — a real firmware link with opt-level 3 + LTO + `--gc-sections` is smaller):

| object | .text | .data | .bss |
|--------|-------|-------|------|
| `vokra-vad-micro` (forward core) | 16,402 B (~16.0 KiB) | 0 | 0 |
| `vokra-core` no_std subset (error + GGUF reader) | 27,333 B (~26.7 KiB) | 0 | 0 |
| **combined upper bound** | **~43.7 KiB** | 0 | 0 |

**Heap** (owned `Vec<f32>` weights — the load-bearing term; `GgufFile::from_external` from a flash/XIP mapping does **not** avoid this, because `weights::vec1d` copies each tensor payload into an owned `Vec<f32>` and `Conv1dW` double-holds `weight` + `weight_t`). Computed from the exact per-tensor shapes and **cross-checked against the committed fixture** (`tests/parity/silero_vad/silero-vad-v5.gguf` = 2,183,424 B = 2,181,128 B of on-disk f32 payload + 2,296 B GGUF header/metadata/padding — exact match):

| binding | f32 count | bytes | ≈ |
|---------|-----------|-------|---|
| both rates + `weight_t` (as loaded today) | 825,602 | 3,302,408 B | **3.15 MiB** |
| 16 kHz only + `weight_t` | 486,785 | 1,947,140 B | **1.86 MiB** |
| 8 kHz only + `weight_t` | 338,817 | 1,355,268 B | **1.29 MiB** |
| both rates, on-disk (no `weight_t`) | 545,282 | 2,181,128 B | 2.08 MiB |

Transient per-frame buffers (reflect pad + conv outputs + encoder intermediates + LSTM matvec temporaries) are O(bins·frames)+O(4·HIDDEN) ≈ a few KiB, freed each frame. The `GgufFile` index (a `BTreeMap` of ~30 tensor names) is a few KiB; the raw GGUF bytes are **borrowed** from the flash image (not copied) — only the weight payloads land on the heap.

**Pass/fail** — Cortex-M55-class MCU on-chip SRAM commonly runs from a few hundred KiB to ~2 MiB (part-dependent):
- **both rates (3.15 MiB): does NOT fit any current M55-class on-chip SRAM** → FAIL on-chip-only.
- **16 kHz only (1.86 MiB): fits only the largest-SRAM parts (or external PSRAM)** → borderline/FAIL.
- **8 kHz only (1.29 MiB): fits parts with ≳1.4 MiB usable SRAM** → PASS on high-SRAM M55, FAIL on smaller.

**Honest verdict: the committed both-rate fixture does NOT fit typical M55 on-chip SRAM as-is; single-rate is borderline.** The static code (~44 KiB) is never the constraint — the weights are. Per the ticket, this WP **records the judgment and stops**; the reductions below are **owner follow-ups, not implemented here**:

- **(a) drop `weight_t` on the MCU build** — the non-transposed `math::conv1d` is the bit-identical reference oracle (`math::conv1d_wt_bitwise_matches_conv1d` pins `==`), so a MCU-only build can carry only `weight` and use `conv1d`, halving the conv-weight heap: saves 280,320 f32 both-rate (1.07 MiB), 177,152 f32 16k (0.68 MiB), 103,168 f32 8k (0.39 MiB). **Precondition**: extend `conv1d_wt_bitwise_matches_conv1d` to Silero's full shapes (it currently pins reduced-size cases) before adopting.
- **(b) bind only the needed rate** (16 kHz for a mic, 8 kHz for telephony) — already an `Option` per rate; a MCU build simply skips the other.
- **(c) zero-copy borrow weights from the flash/XIP mapping** — skip the `Vec<f32>` copy entirely (read f32 in place from XIP flash), dropping the weight heap to ~0. Biggest win, biggest refactor: lifetime-carrying `RateWeights<'a>`, f32 alignment + little-endian guarantees on the mapping, and the M5-14 transpose-on-load interaction.
- **(d)** with (a)+(b)+(c) [16 kHz, no `weight_t`, XIP-borrowed] the persistent heap is ~0 and only a few KiB of transient buffers + the index remain → **fits well under 256 KiB SRAM**. That is the path to the smallest M55 parts; scoped as owner follow-up.

## (4) Owner tasks (T17 / T18) — real Cortex-M55 verification & Helium decision

**T17 (real-device / FVP run).** CC gives you a bit-identical host oracle (§1.2); reproduce it on real Cortex-M55 execution and record RTF / RAM:
- **Paths** (pick one; specifics are yours to name off-public): a real Cortex-M55 devboard; an **Arm Fixed Virtual Platform (FVP)** for the M55 subsystem; or a **QEMU** Cortex-M55 machine model with semihosting.
- **UNKNOWN on this machine**: QEMU for Cortex-M55 is **not installed here** (`which qemu-system-arm` = not found), so the QEMU-semihosting execution path is **unverified** by CC. Installing a dev-side QEMU is not a runtime dependency (zero-dep unaffected), but whether it loads and dispatches the subset is untested.
- **Harness sketch**: build a tiny `#![no_std]` binary that links `vokra-vad-micro` (`--no-default-features`), installs a `#[global_allocator]` (a bump allocator is enough — the forward's allocations are Vec pushes) and a `#[panic_handler]`, feeds a committed fixture frame (or a flash-embedded GGUF via `from_external` over a `&'static [u8]`), and compares `forward_chunk` output to the host `nostd_forward_bit_identical_to_std_*` values. Mind the memory budget (§3) — the both-rate fixture won't fit small parts; use a single-rate GGUF or reduction (a)/(c).

**T18 (Helium / positioning).** MVE intrinsics are **not on stable Rust** (measured: `E0658 stdarch_arm_neon_intrinsics`; `-C target-feature=+mve.fp` "unknown and unstable … might be unsound"; `#[target_feature(enable="mve.fp")]` invalid). So the completion condition is met **scalar**. Options, to decide with T17's real numbers:
- **(i) raw `asm!` MVE** — the rustc/LLVM codegen backend *does* recognise `mve`/`mve.fp` (`--print target-features`), so raw `asm!` MVE is theoretically reachable, the M4-08 RVV-0.7.1 `.insn` precedent (raw words + compile-time `const assert` pin + an asm-grep tripwire). MVE is Thumb2-encoded — more work than RVV. **Only then** add an MVE asm-grep tripwire to `silero-nostd-cross-build.yml`; today there is deliberately none (writing one for instructions we don't emit would be fabrication).
- **(iii) stay scalar (default, safest)** — meets the acceptance criterion, keeps the crate `unsafe`-free.
- (ii) nightly `#![feature(stdarch_arm_neon_intrinsics)]` — **rejected** (stable-only discipline; unstable API can change).

## (5) ESP32-S3 (Xtensa LX7) — CI-excluded, community follow-up

Stock `rustup` has **zero** Xtensa targets (`rustup target list | grep -ci xtensa` = 0) — the esp-rs fork toolchain is required, which stock CI cannot provision. So M5-03's cross-build CI is **thumbv8m only**; ESP32-S3 stays an opt-in / community-maintained tier gated on an esp-rs toolchain. `milestones.md §9` lists Cortex-M55 and ESP32-S3 together, but this target-availability asymmetry is why only the former has a CI leg.

## (7) Tier 3 opt-in / community operating note
