## Read-through notes (apply to every field below)

These three carry over from the decisions that shaped this process. They are
repeated in each record so a reader a year from now does not misread the
numbers.

1. **The automated monitoring workflow was retired (2026-07-04).** The former
   scheduled competitor-changelog workflow (`FR-TL-05`) no longer exists and
   must not be recreated. This manual quarterly review is now the *only*
   monitoring mechanism, which raises the cost of skipping one.
2. **Kill switch C's "Discord active user" sub-threshold is measured by
   proxy.** Discord was not adopted (2026-07-04 / 2026-07-06), so the
   engagement figure is unique participants in GitHub Issues and Discussions
   over the trailing three months. **Note that Discussions is currently
   disabled**, so that channel contributes zero by construction — the proxy is
   effectively Issues-only until that changes (X-05-T23).
3. **Runtime watermarking is deferred, not delivered.** This is tracked as a
   known risk below, not as a completed compliance item. Do not let a green
   engineering column imply the compliance obligation is met.

> **Kill switch C — the agreed threshold is a PASS condition, so the firing
> form is a disjunction.**
> [`quarterly-review-runbook.md`](quarterly-review-runbook.md) states the PASS
> side (§2, line 94: 「star ≥ 500 かつ engagement proxy ≥ 20 を PASS 側とする」;
> restated in §7.2 step 3, line 396), and
> [`scripts/kill-switch-metrics.sh`](../../scripts/kill-switch-metrics.sh)
> computes exactly that:
> `if [ "$STARS" -ge 500 ] && [ "$ACTIVE" -ge 20 ]; then KSC_VERDICT="PASS"`.
> Negating that conjunction gives the firing condition transcribed above —
> stars < 500 **or** participants < 20. Transcribing it as an "and" would
> require *both* figures to be low before C fires, which is a materially laxer
> withdrawal test than the one the project agreed to. Read the verdict off the
> metrics JSON rather than recomputing it by eye: `kill_switch_c.verdict` is
> `PASS` when the switch does **not** fire and `FAIL` when it does (`FAIL` =
> 該当).

> **Kill switch D — which number does the threshold apply to?**
> The metrics snapshot reports **two** contributor counts on purpose:
> `contributors_non_bot_non_cc` (excludes bots and Claude Code) and
> `contributors_excluding_owner` (additionally excludes the maintainer). The
> D wording — "committers other than Claude Code" — reads as the first; DoD
> item 5's requirement that community operation not depend on the maintainer
> alone can only be satisfied by the second. **The two disagree about what
> "3" means, and the choice has not been made.** Record which count you
> applied and why. Do not let the ambiguity resolve itself silently by
> whichever number happens to be quoted.

## Publication

This template lives in the public `docs/governance/` tree, so the default is
that completed review records are published here too — the project's stated
commitment is to develop in the open, and a governance process whose outputs
are private is not verifiable from outside.

**The maintainer decides per record.** If a record must reference private
planning material, cite the conclusion rather than the internal document, so
publishing the record does not require publishing the tree behind it.
