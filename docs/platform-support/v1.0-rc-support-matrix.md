## Part 1 — 「official support」の operational 定義 (T01)

「official support」は**新規発明ではなく既存要件の合成**である。プラットフォーム P が official support であるとは、以下 3 evidence が揃うこと:

| # | evidence 列 | 定義 | 出典 |
|---|------------|------|------|
| (i) | **CI 検証** | NFR-MT-02 の該当 tier で P 向け build/cross-build/test が継続実行され green。required checks は NFR-MT-07。 | NFR-MT-02 / NFR-MT-07 / NFR-PT-01 |
| (ii) | **配布パッケージ** | `docs/deliverables.md` §3.2 の P 行の形態が NFR-MT-08 の CD 自動発行で得られる (手動ビルド配布は不可)。 | NFR-DS-03 / NFR-MT-08 / deliverables §3.2 |
| (iii) | **実機検証記録** | owner の実機テスト記録が `docs/benchmarks/` 系文書に存在する。 | NFR-PF-03/06/08 / X-06 |

**3 列構成の妥当性**: milestones §8 M4-11 完了条件「6 プラットフォームの official support が **CI matrix (NFR-MT-02) と配布パッケージ (NFR-DS-03)** で確認でき、CDN 選定が完了」+ Exit criteria「M4-11 の確認記録」に対応。(i)(ii) は完了条件の明示 2 軸、(iii) は Exit criteria「全プラットフォーム official support 出揃い」の実証裏付け。

**判定対象は 6 プラットフォームのみ** (NFR-PT-01 前段 = Windows / macOS / Linux / Android / iOS / Web (WASM))。サーバ x86-64/ARM64 は desktop Linux/Windows と CI 上同一 evidence で覆われ、RISC-V (`riscv-cross-build`) や IoT 階層 (NFR-PT-02/03) は判定対象外の**補助 evidence** として付録 (Part 6) に記録する (判定に混ぜない)。

**配布パッケージの判定基準を拡大解釈しない**: 完了条件「配布パッケージ (NFR-DS-03) で確認」の判定基準は **deliverables §3.2 のプラットフォーム別形態**であって、NFR-DS-03 が列挙する全 11 形態 (vcpkg/Homebrew/winget/Debian 系まで) の完備ではない。パッケージマネージャ流通は判定対象外の付録チャネル (Part 4 で分離)。

## Part 2 — gap flow 規約 (T01)

確認 matrix は「全部緑になるまで作らない」のではなく、**現状の honest snapshot を先行作成し gap セルを可視化したまま順次緑化する**。赤セルの処理:

1. **owning WP が M4 内に存在** → その WP へ差し戻す (例: Web CI leg = M4-01、Unity WebGL = M4-02、Vulkan 完成 = M4-13)。
2. **owning WP 不在** → issue 起票 + T13 で owner が 3 択判断:
   - (a) M4 内 follow-up ticket を新設、
   - (b) M5 (v1.0 GA) へ送る、
   - (c) 要件側 (deliverables §3.2 / NFR-DS-03) を改訂する。

**gap を残したまま matrix を「確認済」と close しない** (fabricated pass 禁止)。confirm セルは evidence (workflow run URL / 配布物 URL / 実機記録) が存在するものだけ緑。到達していない NFR-MT-02 tier を到達済みと書かない。

**FR-EX-08 の精神**: 「動かないプラットフォームを official と宣言しない」= silent fallback 禁止の配布版。
